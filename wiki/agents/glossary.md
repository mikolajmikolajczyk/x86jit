# Glossary

Emulator / x86 terminology used in this repo. One term per entry. Definitions short; link out for depth.

- **Guest** — the emulated x86-64 program. **Host** — the machine x86jit runs on (x86-64 or ARM64, §1).
- **Lift** — translate decoded x86 instructions into the project's own IR (§7). The lift encodes what each instruction *means*.
- **IR** — intermediate representation: three-address code over temporaries, sitting between iced-x86 and either backend (§6).
- **Temp** — a block-local temporary value in the IR; a monotonic counter (`TempGen`), reset per block (§6.1, §7.2).
- **Val** — an IR operand: a `Temp` or an immediate (`Imm`) (§6.1).
- **Operand lowering** — the layer beneath per-mnemonic lift that reduces an x86 operand (register / immediate / memory) to a `Val`. Memory operands expand to effective-address arithmetic + `Load`/`Store` (§7.1). Load-bearing: nothing lifts without it.
- **Effective address** — `base + index*scale + disp`, plus FS/GS base for segment-prefixed accesses; computed once per operand (§7.1).
- **RIP-relative** — x86-64 addressing relative to the *next* instruction's address. Use iced's computed value, don't recompute (§7 trap 2).
- **Block / basic block** — a straight-line run of instructions ending at the first control-flow op (jmp/jcc/call/ret/syscall/hlt). Unit of lifting, caching, and budgeting (§6.3, §7.3).
- **IrBlock** — a lifted block: ops, `temp_count`, `guest_len`, `icount`, keyed by `guest_start` (§6.3).
- **Backend** — consumer of `IrBlock`. Two exist: **interpreter** (reference, slow, debuggable, the oracle) and **JIT** (Cranelift → host code) (§8).
- **Materialize** — the backend-dependent step turning `IrBlock` into a `CachedBlock` (interpreter wraps `Arc`; JIT compiles) (§8).
- **execute** — the *uniform* step run over a `CachedBlock`, matching on its variant; returns `StepResult` (§8).
- **StepResult** — `Continue` (block done, RIP updated, keep going) or `Exit` (trap out to user) (§8).
- **Exit** — the return-based reason `run()` handed control back: syscall, hlt, unmapped memory, MMIO read/write, unknown instruction, breakpoint, budget exhausted, fault (§5.2).
- **Trap-out** — leaving generated/interpreted code to return an `Exit` to the user. Rare-path only; RAM access never traps out (§1).
- **Translation cache** — guest address → `CachedBlock`. Shared across `Vcpu`s behind an `RwLock`; entries cloned out on lookup (§9).
- **CachedBlock** — cached materialized block: `Interpreted(Arc<IrBlock>)` or `Compiled { entry, guest_len }` (§9.1).
- **CompiledPtr** — raw pointer into the JIT code arena, manually `Send + Sync` (code is read-only + executable from any thread). Introduced in M4 to avoid an M7 rework (§9.1).
- **host_base** — base pointer of the flat guest RAM buffer; the JIT adds `guest_addr` to it to inline RAM access (§8.2.1).
- **ABI (compiled block)** — the fixed signature every compiled block shares: `(cpu: *mut CpuState, mem: *mut MemCtx) -> u64`, where the `u64` encodes a `StepResult`/`Exit` (§8.2.1–8.2.2).
- **Vm** — shared per-machine state: guest memory + translation cache. **Vcpu** — per-guest-thread: `CpuState` + its own `run()` loop. KVM-style split (§2).
- **Dispatcher** — the `run()` loop: fetch/lift, execute, branch on `StepResult`, honor the block budget (§9.2).
- **Budget** — cooperative-scheduling limit for `run()`, counted in **blocks** (§5.1).
- **Differential testing** — run a block natively on an x86 host and compare state against the interpreter; the primary correctness oracle (§13).
- **Oracle** — the reference to compare against. Native execution is the oracle for the interpreter; the interpreter is the oracle for the JIT (§13).
- **SMC** — self-modifying code: a guest write to a page that has translated blocks invalidates them (§10).
- **TSO** — Total Store Order, x86's strong memory model. On weak hosts (ARM), the `MemConsistency` tier emits barriers in codegen so multithreaded guests stay correct (§8.2.3, §11).
- **MemConsistency** — the three-tier speed/strictness knob for ordinary loads/stores on weak hosts (§4.1, §8.2.3): `Fast` (bare STR/LDR — no memory-based sync), `AcqRel` (STLR/LDAPR — the standard TSO mapping, ~99% of correct MT code), `FullTso` (STR+DMB ISH / LDR+DMB ISHLD — the hammer). Escalation ladder per workload; identical code on x86 hosts; locked ops/`mfence` are tier-independent; switching flushes the translation cache. Distinct from `MemoryModel` (address-space layout).
- **LDAPR / RCpc** — ARMv8.3 load-acquire with "processor consistency" (`FEAT_LRCPC`): permits the store→load reordering TSO allows, unlike the stronger `LDAR` (RCsc). What makes the `AcqRel` tier the canonical x86-TSO mapping (§8.2.3).
- **Trap region** — memory tagged `RegionKind::Trap`; every access yields `Exit::MmioRead`/`MmioWrite` instead of touching RAM (§4.2).
- **FlagMask** — which of CF/PF/AF/ZF/SF/OF an IR op updates. Replaces a naive `set_flags: bool` because x86 flag updates aren't uniform: `inc`/`dec` keep CF, logic ops force CF=OF=0, shifts update only when count ≠ 0 (§6.2).
- **GetCond** — IR op materializing a `Cond` as a 0/1 value (flags-as-data), for `setcc`/`cmovcc`/`adc`-`sbb` lowering/`rcl`-`rcr`. Distinct from `Branch`, which only *consumes* a condition (§6.2).
- **Instruction atomicity (pitfall #0)** — the rule that no effect of a guest instruction may commit before its last possible trap, so the RIP-retry convention can safely re-execute it. Trapping ops (load/store) emitted before committing ops (WriteReg, flags) (§7 pitfall 3, §16).
- **Interior mutability (of guest RAM)** — `Memory` is written through `&self` (`UnsafeCell` + hand-written `unsafe impl Sync`), because one buffer is shared and concurrently written by many vcpus, ordered by TSO barriers not by Rust `&mut` (§8).
- **Backend (injected)** — the JIT/interpreter is a `Box<dyn Backend>` passed into `Vm::with_backend`, not a config enum; the core cannot name the downstream `x86jit-cranelift` crate (§4.1, §8).
- **Long mode** — 64-bit x86-64 mode; the *only* mode this library supports. Decoder always 64-bit, segments (except FS/GS) base 0, flat addressing (§1, §17).
- **Seam** — a cheap, deliberately-left extension point (a parameter instead of a constant) that keeps a future option open **without** building a framework for it now. §17 defines three: mode as a value (not the literal `64`), mode in the cache key, and the single `effective_address` choke-point. The opposite of *machinery* — abstractions/traits/config built speculatively before a second implementation exists, which §17.6 forbids.
- **CpuMode** — the seam-1 mode enum; today only `Long64`. Makes the decoder and address computation *ask* for the mode instead of hardcoding 64-bit (§17.3).
- **BlockKey** — the seam-2 cache key sketch, `(guest_addr, mode)`; today `mode` is always `Long64`, so the live cache keys on `u64` alone (§17.4).
